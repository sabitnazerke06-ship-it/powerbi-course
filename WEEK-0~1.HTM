01

Your Excel is already a model — you just did not call it that

If you have spent years in audit or FP&A, you have built data models. You called them spreadsheets. The concepts are the same — they just live in different tools and scale very differently.

Think about a typical P&L model in Excel. You have a tab for actuals pulled from the GL. Another tab for budget. Another that pulls from both and calculates variances. Each tab has rows and columns. Some columns reference other tabs. There are lookup formulas that match entities across sheets. You are already thinking in tables, references, and relationships.

What changes in Power BI is that these relationships become explicit, named, and governed. Instead of a VLOOKUP that happens to join two tables by product code, you declare a relationship between them once — and every measure, every visual, every filter in the report uses it automatically. And it works on millions of rows without reformulating every column.

From Excel to Power BI — the same concepts, different tools

The key shift is this: in Excel, you hold the model together with formulas. In Power BI, the model holds itself together — and you build measures on top of it. This is not just a tool difference. It is a fundamentally different way of thinking about data.

02

Tables and relationships

What a table is

A table is a structured collection of data where every row represents one observation of the same thing, and every column represents one attribute of that thing. Every row must have the same columns. A cell can only contain one value.

This sounds obvious, but it rules out a lot of what people do in Excel: merged cells, header rows mid-sheet, columns that mean different things depending on the row, values like "Q1 / Q2 combined." None of these work in a data model. The table must be consistent, rectangular, and unambiguous.

What a well-structured table looks like — vs what does not work

What a relationship is

A relationship connects two tables through a shared column — a key. One table has the key as its primary identifier (the "one" side). The other table references that key in many rows (the "many" side).

Think of it this way: if you have a table of orders and a table of products, every order references a product. Many orders can reference the same product. The product exists once in the products table, and many times in the orders table. That is a relationship — specifically, a one-to-many relationship.

03

One-to-many — the workhorse relationship

The one-to-many relationship is the foundation of almost every data model you will build. One row in table A relates to many rows in table B. The "one" side is always a dimension or reference table. The "many" side is always a fact or transaction table.

One-to-many relationship — how filters flow through the model

Notice the direction of the arrow: filters flow from the "one" side (Products) to the "many" side (Orders). If you select "Stapler" in a slicer, Power BI finds all rows in Orders where ProductID = P-001. This is how filtering works throughout the model. You declare the relationship once, and it works everywhere.

The "one" side must have unique values. Products must have exactly one row per ProductID. If it has duplicates, Power BI will not allow the relationship — or will make it many-to-many, which causes different problems (covered next).
04

Many-to-many — the problem you must avoid

A many-to-many relationship occurs when the key column has duplicates on both sides of the relationship. Power BI will warn you and either refuse the relationship or downgrade it to many-to-many cardinality — either way, stop and fix the duplicates on the dim side. Do not click through and accept it.

The most common cause: you are trying to use a table that is not yet a proper dimension table — it still has duplicate keys on what should be the "one" side.

Why many-to-many produces wrong numbers

The fix is always the same: deduplicate the "one" side. Create a proper dimension table that has exactly one row per unique key value. Your fact table can then reference it many times. This is what Power Query is for — building clean, unique dimension tables from messy source data.

If Power BI warns you about a many-to-many relationship — stop. Do not click through and accept it. Go back and find the duplicate keys on your dimension side. The warning is telling you that your model will produce wrong numbers.
05

The star schema

The star schema is the most widely used data modeling pattern in business intelligence. It gets its name from its shape: a central fact table surrounded by dimension tables, each connected by a one-to-many relationship. When you draw it out, it looks like a star.

It was developed as the solution to a specific problem: analytical queries on relational databases were slow and complex when data was normalised across many tables. The star schema trades storage efficiency for query speed and simplicity. For Power BI, it is essentially the only model structure that performs well at scale.

Star schema — Global Super Store example

Every dimension table connects to the fact table. Filters flow inward from dimensions to facts — never the other way (in a basic star schema). This single-direction filter flow is what makes the model predictable and performant.

The Global Super Store dataset you will use this week already follows this structure. Part of Week 2 is building it — which means you will understand exactly why each table exists and what role it plays.

06

Fact vs dimension tables — what goes where

This distinction is the single most important concept in data modeling. If you put something in the wrong type of table, the model produces wrong numbers or becomes impossible to maintain.

Fact table

Contains: measurements, quantities, amounts — things you add up.

Rows represent: one transaction or event (one order line, one payment, one daily balance).

Typical columns: foreign keys to dimensions + numeric measures.

Grows constantly: every new transaction adds a row. Millions of rows is normal.

Examples: Orders, Payments, LCR inflows, LCR outflows

Dimension table

Contains: descriptive attributes — things you filter and group by.

Rows represent: one entity (one product, one customer, one date).

Typical columns: primary key + text/category attributes.

Stable: grows slowly. Often thousands of rows, but in finance contexts dims can be hundreds of thousands or even millions — DimCustomer at a retail bank, DimSecurity at an asset manager.

Examples: Products, Customers, Date, Geography, Counterparty

Size is not what defines a dimension. What defines it is the role: dimensions are used for filtering and grouping, facts for aggregation. A dim with three million rows is still a dim. What matters is that it grows slowly and has one row per entity.
How to identify fact vs dimension in practice

A practical test: can you SUM it meaningfully?

Fact table columns should be things you can sum and get a meaningful answer: total sales, total quantity, total outflows. If summing a column produces nonsense — like summing customer ID numbers — it belongs in a dimension.

There is one exception: foreign keys in the fact table are numbers that look like they could be summed but should not be. They are identifiers, not measures. A well-built model hides foreign key columns from report view so users cannot accidentally drag them into visuals.

From relational database to data warehouse

Understanding where the star schema came from helps you understand why it is designed the way it is — and why data engineers structure source systems differently from analytical models.

How data modeling evolved — and where Power BI fits

The key insight from this diagram: you work at the Power BI layer. By the time it reaches you, it should already be structured for analytics. 08

Reading an unfamiliar dataset

Every engagement starts with a dataset you have not seen before. Knowing how to quickly orient yourself — identify fact tables, find dimension candidates, understand the grain — saves hours of confusion.

The grain: the most important question about any table

The grain of a table is what one row represents. It is the most fundamental thing to establish about any table. Before you do anything else with a dataset, answer this question for every table in it.

Grain examples — Global Super Store

Orders table: one row = one order line item (one product in one order)

Products table: one row = one product

Date table: one row = one calendar day

LCR inflows: one row = one inflow category for one legal entity on one reporting date

If you cannot state the grain in one clear sentence, the table is not ready to model. Either it mixes different grains (bad source data), or you do not yet understand it (talk to the data engineer).

A four-step process for reading a new dataset

  • Step 1 — Count the rows. A table with 5 million rows is almost certainly a fact table. A table with 500 rows is almost certainly a dimension.
  • Step 2 — Find the primary key. Which column (or combination of columns) uniquely identifies each row? That column is the key. If no column is unique, the table probably needs transformation.
  • Step 3 — Identify the numeric columns. These are your measures. Are they additive (you can sum them)? Semi-additive (sometimes)? Non-additive (never)? This determines how you will build measures later.
  • Step 4 — Look for foreign keys. Columns with IDs that appear to reference another table are foreign keys. These are where relationships will connect.
When in doubt, ask the data engineer. They built the pipeline that produced this data. They know what a row represents, what the keys mean, and where the data came from originally. A 20-minute conversation at the start of a project saves days of reverse-engineering later.

01 / 07Multiple choice
In a star schema, which table should always have unique values in its primary key column — and why?
Correct. The dimension table is the "one" side of the relationship. If its key column has duplicates, Power BI cannot create a valid one-to-many relationship — it either refuses the relationship or creates a many-to-many, which produces incorrect totals. The fact table's primary key (OrderID, etc.) is also unique per row, but the critical constraint for relationships is on the dimension side.
Not quite. The answer is B. The dimension table is the "one" side of every relationship. Its primary key must be unique. If it is not, Power BI creates a many-to-many relationship and your aggregations will produce wrong numbers.
02 / 07Multiple choice
You receive a table from the data engineer called "ProductSales" with columns: ProductID, ProductName, Category, SalesAmount, Quantity, OrderDate. What is wrong with this table from a modeling perspective?
Correct. ProductName and Category describe the product — they belong in a dim_Products table. SalesAmount, Quantity, and OrderDate relate to the transaction — they belong in a fact table. Mixing them creates a denormalised flat table. You can use it, but you lose the ability to filter products independently, manage product attributes in one place, or build a proper date table relationship. Split them.
Not quite. The answer is C. Fact attributes (amounts, quantities, transaction dates) and dimension attributes (product name, category) are mixed in the same table. This should be split into a fact table (OrderID, ProductID, SalesAmount, Quantity, OrderDate) and a dimension table (ProductID, ProductName, Category).
03 / 07Multiple choice
In your Power BI model, filters flow from a dimension table to a fact table. A user selects "East" in a Region slicer. What happens?
Correct. This is exactly how relationships work. The slicer filters the dimension table to "East." That filter propagates through the one-to-many relationship to the fact table, restricting it to rows where GeographyID matches East. Every measure — SUM(Sales), COUNT(Orders), etc. — recalculates on that filtered set. This is the engine behind every slicer and filter in Power BI.
Not quite. The answer is B. The filter flows from the dimension (one side) through the relationship to the fact table (many side). The fact table is filtered to rows that match the selected value, and all measures recalculate accordingly. This is how every slicer in Power BI works.
04 / 07Scenario
You are given the Global Super Store dataset as a single flat CSV file with all columns in one table. What should you do before building any measures or visuals?
Correct. A flat file loaded as a single table is not a data model — it is a flat table with no relationships. Every measure you write on it will be working against an unstructured blob of data. Use Power Query to extract unique dimension tables (one row per product, one row per customer, one row per date) and a clean fact table with foreign keys. Then define the relationships. This is Week 2's exercise.
Not quite. The answer is C. A flat file needs to be restructured into a star schema before modeling. Power Query lets you extract unique dimension tables and create a clean fact table with foreign keys. Building on a flat table without relationships will cause incorrect results as soon as you try to filter across entities.
05 / 07Multiple choice
You are looking at a table with 8 million rows. Column A contains dates, Column B contains product codes, Column C contains decimal amounts. What is this table most likely, and what should you check first?
Correct. 8 million rows almost certainly means a fact table — most dimensions are much smaller (though customer or security dims in finance can also be large). The first question is always the grain: what does one row represent? Is it one transaction? One daily position per product? Once you know the grain, you can identify the primary key (the combination of columns that uniquely identifies each row) and start thinking about which dimensions it connects to.
Not quite. The answer is B. 8 million rows strongly suggests a fact table. The first thing to establish about any table is its grain: what does one row represent? From there, you identify the primary key and plan the relationships.
06 / 07Open-ended
Draw or describe in words the star schema you would build for the LCR dataset. What would the fact table contain? What dimension tables would surround it? What would be the grain of the fact table?
Model answer Fact table: fact_LCR_Flows Grain: one row = one flow category for one legal entity on one reporting date. Columns: DateID (FK), EntityID (FK), ProductTypeID (FK), CounterpartyID (FK), FlowAmount, WeightedAmount, MaturityBucket. Dimension tables: dim_Date (DateID, Date, Year, Quarter, Month) · dim_Entity (EntityID, LegalEntityName, Jurisdiction) · dim_ProductType (ProductTypeID, Category, RegulatoryCode, FlowDirection: inflow/outflow) · dim_Counterparty (CounterpartyID, CounterpartyName, CounterpartyType: retail/wholesale/financial). The star schema keeps regulatory codes in dim_ProductType so you can filter by them without cluttering the fact table. All four dimensions connect to the fact table via one-to-many relationships.
07 / 07Open-ended
Your colleague says: "I just loaded the Global Super Store CSV as one table and built my whole report on it. It works fine!" What would you tell them — and what specific problems might appear later?
Model answer It works fine — until it does not. Three specific problems that will appear: (1) Duplicate dimension data. The product name, category, and subcategory are repeated on every order row. If someone fixes a typo in a product name, they must update every row. In a dimension table, they update one row. (2) Incorrect totals when filtering across entities. Without proper relationships, a slicer on Category may not correctly filter a measure calculated on a different aspect of the same flat table — the model has no way to understand the relationship between attributes. (3) Performance. A flat 10,000-row table works. A flat 10 million-row table with repeated text strings for every attribute will be slow. The star schema compresses fact tables because they only store IDs, not repeated text. The flat file approach delays these problems — it does not solve them.
