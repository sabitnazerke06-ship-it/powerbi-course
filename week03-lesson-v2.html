01

Implicit vs explicit measures — always choose explicit

When you drag a numeric column from the Fields panel into a visual, Power BI automatically aggregates it — usually as a SUM. This is called an implicit measure. It is convenient for quick exploration and functionally correct for simple cases. It is also something you should stop relying on immediately.

An explicit measure is a DAX expression you write yourself, stored in the model, with a name you chose. It lives in the model permanently, can be reused across any visual or report page, and its logic is visible to anyone who opens the file.

Implicit measure (drag & drop)

Name: "Sum of SalesAmount" — auto-generated

Location: exists only in that one visual

Reusability: none — you recreate it in every new visual

Logic visible? No — just "SUM" applied to a column

Use when: quick exploration only, never in a client report

Explicit measure (DAX)

Name: "Total Revenue" — you chose it

Location: stored in the model, available everywhere

Reusability: drag into any visual, any report page

Logic visible? Yes — click it to see the full DAX expression

Use when: always, for any number shown to a client

The practical rule: every number that appears in a client report should be an explicit measure with a name a business user can read. "Sum of SalesAmount" is not a measure name. "Total Revenue" is. "Net Margin % (excl. promotions)" is even better — it tells you exactly what it represents and removes ambiguity.

How to create a measure in Power BI Desktop: Click on any table in the Fields panel, then go to Home → New Measure. The formula bar opens. Type your DAX there. The measure appears in that table with a calculator icon (∑). You can then drag it to a different table — typically a dedicated _Measures table (covered in Part 6).
02

DAX functions — organised by what they do

Rather than presenting functions as a flat list, it helps to see them grouped by purpose. Every function below belongs to one of three jobs: aggregating numbers, counting things, or dividing safely.

Before writing any measure, think in Excel terms first. You already know how to sum a column in Excel (=SUM(B:B)) and how to count unique values (=COUNTIF). DAX works on the same logic — the key difference is that instead of a cell range, you provide a table column, and DAX automatically respects whatever filter is currently active on the report. The measure does not change; the filter context around it changes.
Group 1 — Aggregation
Total Revenue
Total Revenue = SUM(FactSales[SalesAmount])
Sums SalesAmount across every row that survives the current filter. The most common measure you will write.
Total Cost
Total Cost = SUM(FactSales[TotalCost])
Same pattern. Write this as a separate measure so you can reuse it in Gross Profit and Gross Margin %.
Gross Profit
Gross Profit = [Total Revenue] - [Total Cost]
References other measures — not columns. Keeps logic centralised: if Total Revenue changes, Gross Profit updates automatically.
Avg Unit Price
Avg Unit Price = AVERAGE(DimProduct[UnitPrice])
AVERAGE over a column. Useful for product pricing analysis — respects whatever product filter is active.
Group 2 — Counting
Total Transactions
Total Transactions = COUNTROWS(FactSales)
Counts every row in the filtered fact table. Use COUNTROWS for transaction counts — it is explicit and skips nothing.
Unique Customers
Unique Customers = DISTINCTCOUNT(FactSales[CustomerKey])
Counts unique values — one customer who bought 10 times counts as 1. Use when "how many" means unique entities, not row count.
COUNTROWS vs COUNT — why it matters

COUNTROWS(FactSales) counts every row in the table, regardless of column values. It cannot skip rows.

COUNT(FactSales[OrderNumber]) counts only rows where OrderNumber is non-blank. If any rows have a blank OrderNumber — due to a data quality issue — COUNT gives a lower result than expected, silently understating your transaction volume.

Rule: use COUNTROWS for reliable row counts. Use COUNT only when you specifically want to count non-blank values in a column.

Group 3 — Safe Division (DIVIDE)
Gross Margin %
Gross Margin % = DIVIDE([Gross Profit], [Total Revenue])
Always use DIVIDE for ratios. Handles division-by-zero gracefully. References measures, not raw columns.
Avg Revenue per Transaction
Avg Rev per Transaction = DIVIDE([Total Revenue], [Total Transactions])
Divides two measures. Gives average revenue per transaction line — adapts to whatever filter context is active.
How SUM responds to filter context — the core concept

This is the entire concept of filter context in one diagram. SUM(FactSales[SalesAmount]) always sums whatever rows are currently visible. When a slicer selects "Online," the filter reduces the visible rows — and SUM recalculates. The measure does not change. The context around it changes. This understanding is the foundation of everything that follows in DAX.

03

Why DIVIDE — not the slash operator

Both of the following expressions produce a gross margin percentage. Only one is safe to use in a production model:

-- Dangerous: the / operator
Gross Margin % = [Gross Profit] / [Total Revenue]

-- Safe: DIVIDE
Gross Margin % = DIVIDE([Gross Profit], [Total Revenue])

What happens when a filter produces zero revenue — a product that was returned, a future date with no transactions, a new store not yet open? The / operator returns a division-by-zero error. That error propagates into every visual using the measure, breaking the report for the client.

DIVIDE handles this case gracefully. When the denominator is zero or blank, it returns blank by default — the cell shows empty rather than an error. You can also provide a custom fallback value as a third argument:

-- Returns blank when no revenue (default behaviour)
Gross Margin % = DIVIDE([Gross Profit], [Total Revenue])

-- Returns 0 instead of blank — use when blank would confuse stakeholders
Gross Margin % = DIVIDE([Gross Profit], [Total Revenue], 0)
Always reference measures, not columns, in calculations. DIVIDE([Gross Profit], [Total Revenue]) divides the aggregated totals and respects filter context. DIVIDE(FactSales[SalesAmount], FactSales[TotalCost]) attempts to divide column by column in row context — which is not how measures evaluate. Reference the measures you have already built.
04

Working backwards from a visual to the measure

A stakeholder rarely says "I need a DIVIDE measure with a CALCULATE in the denominator." They say things like "show me gross margin by category" or "I want to see how online revenue compares to store revenue over time." Your job is to decompose that requirement into its parts.

Use this four-part decomposition every time:

  • Measure: What number goes on the axis? What is being quantified?
  • Dimension: What dimension drives the rows or the X-axis?
  • Filter: Is there a restriction — a specific segment, a time period, a channel?
  • Visual type: Does this show comparison, trend, composition, or a single value?
Example decomposition — "Gross margin by product category for Contoso"

Practice this decomposition on every requirement before opening Power BI. Once you can name the measure, the dimension, the filter, and the visual type, writing the DAX is straightforward.

05

Choosing the right visual

Visual choice is not aesthetic preference — it is a communication decision. The wrong visual type obscures the insight even when the underlying measure is correct.

📊
Bar / Column chart
Comparing values across categories. Use when the question is "which is bigger?"
Avoid: more than ~12 categories, or when showing trend over time.
📈
Line chart
Trends over time. Use when the X-axis is a continuous time dimension.
Avoid: non-time X-axis, categorical comparisons.
🔢
Card / KPI
Single summary number. Use for Total Revenue, Gross Margin %, or any headline KPI.
Avoid: when context matters — a number without dimension is often misleading.
🥧
Pie / Donut chart
Part-to-whole at a glance with very few slices (2–4).
Avoid: more than 4 slices, time series, negative values.
📋
Table / Matrix
Exact values with multiple dimensions. Use when the audience needs to look up specific numbers.
Avoid: as a default — tables hide patterns that charts reveal.
🗺️
Map visual
Geographic distribution. Use when location is the key dimension.
Avoid: when geography is not meaningful to the analysis.
Financial audiences love Cards. Before building any dashboard for a finance team, add a row of Card visuals at the top — Total Revenue, Gross Profit, Gross Margin %, and one KPI relevant to the client's business. These are the numbers an executive reads first. Everything else on the page provides context to those cards.
06

The measure table pattern

By default, when you create a measure, Power BI stores it in whichever table you had selected. In a model with 50 measures spread across 8 tables, finding any particular measure becomes a search exercise. The solution is a dedicated measures table.

Creating the _Measures table

In Power BI Desktop: go to the Modeling tab → click "New Table" → type the following DAX, then press Enter:

-- Create an empty placeholder table
_Measures = DATATABLE("x", INTEGER, {{0}})
-- Hide the "x" column by right-clicking → Hide in report view

Now move all your measures into this table by right-clicking each measure → Edit → change the Home Table dropdown to _Measures. The underscore prefix causes it to sort to the top of the Fields panel.

Display folders — organising within the table

Once you have more than ~10 measures, use display folders to group them. Select a measure → Properties pane → type a folder name in the Display Folder field. Examples: Revenue, Cost & Margin, Counts, Ratios. Measures in the same folder name are automatically grouped in the Fields panel.

Self-documenting models. A _Measures table with display folders is one of the clearest signals to a client or colleague that a model was built professionally. When they open the file, every calculation is in one place, clearly named and organised. It reduces onboarding time from hours to minutes.
07

BLANK — not zero, not an error

Power BI distinguishes between three different states for a measure cell: a number (including zero), an error, and blank. Blank is its own value — it means "there is no result for this combination." Understanding blank is essential because it behaves differently from zero in calculations and visuals.

Where blank comes from

The most common source: filter combinations that produce no matching rows. If you ask for "Online channel sales in a category that has never had an online sale," the result is blank — not zero. There were no transactions, so there is no sum to display.

-- DIVIDE returns blank when denominator is zero or blank
Gross Margin % = DIVIDE([Gross Profit], [Total Revenue])
-- If Total Revenue = 0 or blank → result = blank (not an error)

-- You can test for blank explicitly
Margin Display =
  IF(
    ISBLANK([Total Revenue]),
    "No sales",
    DIVIDE([Gross Profit], [Total Revenue])
  )

Blank ≠ zero in arithmetic

In DAX, blank propagates through arithmetic in a specific way: BLANK() + 0 = 0, but BLANK() * anything = BLANK(). This asymmetry surprises people. The safe pattern: always use DIVIDE (which returns blank safely) and use ISBLANK() when you need to distinguish "no data" from "data that equals zero."

A blank cell in a visual is not a bug. When a client asks "why is this cell empty?", the answer is usually: there were no transactions for that combination of filters. Blank is the correct representation of "no data" — it is preferable to showing zero, which would imply that something happened and resulted in a zero, which is a different business situation entirely.
08

Three mistakes to avoid from day one

These are the most common errors in early-stage DAX work. Each one produces results that look correct but are subtly wrong — the kind that only get discovered when a client checks the numbers against their source system.

1
Using / instead of DIVIDE

[Gross Profit] / [Total Revenue] will return an error when Revenue is zero or blank in any filter context — a new product, a future date, a filtered-out channel. That single error cell breaks every visual using the measure. Always use DIVIDE.

2
Using COUNT instead of COUNTROWS for row counts

COUNT(FactSales[OrderNumber]) silently skips rows where OrderNumber is blank. If your source data has any blank keys — which is common in imported datasets — you are understating your transaction count. COUNTROWS counts every row with no exceptions.

3
Dividing columns directly instead of measures

DIVIDE(FactSales[SalesAmount], FactSales[TotalCost]) in a measure body is a compilation error — measures have no row context, so column references without an aggregator are invalid. Write DIVIDE([Total Revenue], [Total Cost]) — reference the explicit measures you already built.

06 / 07Open-ended
Using the Contoso dataset, a stakeholder asks: "How many unique products did we sell in each product category, and what percentage of our total product catalogue does that represent?" Write the two DAX measures this requires.
Model answer Measure 1 — Products Sold Products Sold = DISTINCTCOUNT(FactSales[ProductKey]) This counts unique products that appear in the filtered sales. When the visual is filtered to a category, it shows only products actually sold in that category.

Measure 2 — % of Catalogue % of Catalogue = DIVIDE([Products Sold], COUNTROWS(DimProduct)) COUNTROWS(DimProduct) gives the total number of products in the catalogue — it is not filtered by category because we reference the dimension table directly, which is not affected by the product filter propagated from FactSales. DIVIDE handles the zero-denominator case safely.
