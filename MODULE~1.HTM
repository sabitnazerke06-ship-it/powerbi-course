01

What Power Query actually does — and why it matters

Every Power BI report has a hidden layer that most people never think about: the place where raw data gets transformed before it becomes the clean tables in your model. That layer is Power Query. Get it wrong and no amount of clever DAX will save you. Get it right and your measures are simple, your model is fast, and your data is auditable.

Power Query is an ETL tool (Extract, Transform, Load) built into Power BI. It connects to your source data, applies a sequence of transformations, and outputs clean tables that are loaded into the Power BI data model. Critically: Power Query transformations happen at refresh time, before DAX ever runs. The data DAX sees is already clean.

The ETL pipeline — Power Query's position in the stack

Power Query applies its steps in a recorded sequence — every click in the editor is translated into an M language step that can be reviewed, edited, and replayed. This means your data preparation is reproducible, documented, and auditable — critical for regulatory reporting where you need to explain every transformation applied to source data.

The financial services principle: Every transformation applied to source data before it reaches a report must be documented and reproducible. Power Query's Applied Steps pane is your transformation audit trail. Clients and auditors can ask "why does this number differ from the source?" — your Power Query steps are part of the answer.
02

Tour of the Query Editor — find your way around

Open Power Query from Power BI Desktop: Home → Transform data. A new window opens — that's the Power Query Editor. The whole window has five things you will use every day. The mockup below shows where each one lives, with a numbered guide underneath.

Power Query Editor — anatomy
1
Ribbon. Every transformation lives here, grouped by tab. Home has the most-used actions (filter, sort, change type, remove columns, merge). Transform changes existing columns. Add Column creates new ones. View opens the Advanced Editor and toggles column profiling.
2
Queries pane. Every query (table) you have loaded. Click a name to switch the preview. Naming convention: model tables in PascalCase or with prefixes (fact_InternetSales, dim_Product); helper queries you do not load to the model get a leading underscore (_staging_orders) so they're visually separate.
3
Formula bar. The M expression for the currently selected step. You can edit it directly here for precision tweaks the UI doesn't expose — e.g., add a third column to a renames list without re-opening the dialog.
4
Data preview. The data after the currently selected step is applied. First 1,000 rows only — the full transformation runs at refresh. Watch the data-type icon in each column header (123, ABC, calendar). If a number column shows ABC, your aggregations will silently fail.
5
Applied Steps. Every transformation as a recorded step, in order. Click any step to time-travel — the preview rewinds to that point. Right-click → rename a step (e.g., FilteredOutReturns reads better than Filtered Rows1). The ✕ on a step deletes it.
Important: Power Query is a preview environment. Changes do not affect the loaded data until you click Close & Apply (top-left ribbon button). If you close without applying, your unsaved changes are discarded.
03

Core transformations — the ones you use every week

Nearly all your Power Query work happens through eight transformations. You learn them by clicking the buttons — the M code is written for you. The "Where to find it" hint shows which ribbon tab.

🔤
Change data type
Set a column's type: Whole Number, Decimal, Text, Date, True/False. Always the first thing to check after loading. Numeric columns imported from CSV default to Text — every calculation fails silently until you fix this.
Where: click the type icon in the column header, or Home → Data Type
✂️
Remove rows / Keep rows
Filter rows by condition, remove blanks, remove errors, remove top/bottom N rows. "Remove top rows" is often needed to skip the metadata header rows in financial system exports.
Where: Home → Remove Rows · or click the column filter dropdown
📋
Rename columns
Rename cryptic source columns to your naming convention (ORDNO → OrderID, AMT → SalesAmount). Column names in Power Query become column names in DAX — fix them here once, not in every formula.
Where: double-click the column header, or Transform → Rename
🗑️
Remove / Choose columns
"Remove Columns" drops what you select. "Choose Columns" keeps what you select. Drop unused columns aggressively — every column in the model costs memory and refresh time, even if no visual references it.
Where: Home → Remove Columns · or right-click the header
🔠
Add column — Custom / From examples
Create a new column from existing ones — string concatenation, date extraction, classification. "From examples" lets you type the result you want and Power Query infers the formula. Complex business logic belongs in DAX, not here.
Where: Add Column → Custom Column · or From Examples
🔄
Replace values
Replace a specific value in a column with another. Used for cleaning: replacing "N/A" with null, standardising inconsistent labels ("On-line" → "Online"), fixing encoding artefacts in text columns.
Where: right-click a value in the preview → Replace Values
📊
Group By
Aggregate rows by one or more columns — equivalent to SQL GROUP BY. Useful for pre-aggregating large tables before loading (daily transactions rolled up to monthly), or for building a small dimension out of a fact column.
Where: Home → Group By · or Transform → Group By
🔃
Unpivot columns
Convert column headers into rows — the opposite of pivot. Essential for normalising wide financial exports where each column is a period (Jan, Feb, Mar…) into a long format (Date, Value) that Power BI can chart.
Where: select columns → Transform → Unpivot Columns
The division of labour: Power Query = data shaping (types, renames, dropping rows/columns, splitting, combining). DAX = business logic (measures, calculations, time intelligence). When in doubt, ask: "does this transformation depend on what the user clicked in a slicer?" If yes, it's DAX. If no, it's Power Query.
04

M language — just enough to read it

Every action you take in Power Query — every click on a button, every change in a dialog — generates a line of M code behind the scenes. You almost never need to write M from scratch. But you do need to read it well enough to (a) understand what an inherited query does and (b) make small tweaks in the formula bar that the UI doesn't expose.

The structure of an M query

Every Power Query query is a single let ... in expression. Inside let, you define a sequence of named steps; in declares which step is the final output. Here's a typical short query for fact_InternetSales:

let
    // 1 — connect to the source
    Source = Csv.Document(File.Contents("C:\aw_data\FactInternetSales.csv"),
        [Delimiter=",", Encoding=65001, QuoteStyle=QuoteStyle.None]),

    // 2 — first row becomes column headers
    PromotedHeaders = Table.PromoteHeaders(Source, [PromoteAllScalars=true]),

    // 3 — set explicit data types
    ChangedType = Table.TransformColumnTypes(PromotedHeaders, {
        {"SalesOrderNumber", type text},
        {"OrderDateKey", Int64.Type},
        {"SalesAmount", type number}
    }),

    // 4 — keep only rows with a positive order quantity
    FilteredRows = Table.SelectRows(ChangedType, each [OrderQuantity] > 0)

in
    FilteredRows   // the final output of this query

The one thing to notice: each step takes the previous step's result as input. FilteredRows references ChangedType; ChangedType references PromotedHeaders. Reordering steps can break the script — if a later step references a column that doesn't yet exist at that point, you get an error.

When to open the Advanced Editor: The formula bar shows the M for the currently selected step. The Advanced Editor (View → Advanced Editor) shows the full let ... in query in one text view. Open it for code review, to copy a query into another file, or to make a precise edit the UI cannot do — like reordering a list of column renames without re-clicking through the dialog. For everything else, the UI is faster and writes the M for you.
05

Merge & Append — combining data from multiple sources

Two operations combine queries: Merge (equivalent to SQL JOIN) and Append (equivalent to SQL UNION).

Merge (JOIN) — add columns from another table

Merge joins two queries on a matching column and returns a table with columns from both. You choose the join type: Left Outer, Right Outer, Inner, Full Outer, Left Anti, Right Anti. The join columns must match in type and format — text "1001" ≠ number 1001.

Click Home → Merge Queries, pick the two queries, click the join column in each. Power Query adds a new column containing the matched table; click its expand icon (⇆) to bring in the columns you want.

Merge vs Relationship — which to use? In most cases, do not merge dimension data into fact tables in Power Query. Instead, load them as separate tables and create a relationship in the semantic model. Relationships are dynamic and efficient — the fact table stays compact and the dimension can be shared across multiple fact tables. Merge in Power Query only when you need a column for a transformation step inside the query, not as the final shape of the model.

Append (UNION) — stack rows from multiple tables

Append stacks rows from two or more queries into one table. The column names must match (or unmatched columns become null). The common use case: combining monthly files into one fact table, or unioning the same transaction type from different source systems.

Click Home → Append Queries, pick the queries to combine, click OK. The result is one query with all rows stacked in the order you appended.

06

What is a semantic model?

After Power Query finishes cleaning and shaping your data, you have a set of tables loaded into the Power BI data model. The semantic model is the layer on top of those tables that makes them understandable to the people who will build reports and ask questions.

A semantic model consists of: the tables and columns (loaded by Power Query), the relationships between tables, the measures (DAX calculations), column display names and descriptions, data formats and categories, hierarchies (Year → Quarter → Month), and Q&A synonyms.

The semantic model sits between your raw data and every report, dashboard, or natural-language question built on it. A well-built model makes all reports correct by default. A poorly built one means every report builder has to work around the same problems independently.

The semantic model as a single source of truth. In financial services, the same metric (e.g., "net revenue") should have exactly one definition — the measure in the semantic model. If two report builders calculate it differently, one is wrong. A semantic model with documented, tested measures enforces consistency across all reports built on it.
07

Star schema design

The star schema is the recommended data-model pattern for Power BI. It is named for its shape: one or more fact tables in the centre, surrounded by dimension tables that radiate outward like star points.

AdventureWorks — star core (with two snowflake chains)

This is the model you'll build in the exercise: 1 fact table + 5 directly-linked dimensions, plus two snowflake chains (Customer → Geography, Product → Subcategory → Category) where one dim links to another. The star shape works because Power BI's query engine is optimised for it: dimension tables are small (tens to hundreds of rows), fact tables are large (thousands to millions). Filters on small dims propagate to the big fact via relationships — so the engine filters most of the work using the small tables.

08

Relationships — how tables connect

Relationships are the connections between tables in the semantic model. You create them in Model view by dragging a line between two key columns that contain matching values. Three properties matter for every relationship:

1. Cardinality — how many rows match?

One-to-Many (1:*) — the standard. One row in the dimension matches many rows in the fact. dim_Product has one row per product; fact_InternetSales has many rows per product. Always this type between dim and fact.

One-to-One (1:1) — rare. Used when splitting a table for performance or security.

Many-to-Many (*:*) — problematic. Usually a missing bridge table or a design problem. Avoid unless you know exactly why.

2. Filter direction — which way do filters flow?

Single (recommended): filters flow from the "one" side (dim) to the "many" side (fact). dim_Product filters fact_InternetSales — selecting "Bikes" shows only Bikes sales. The fact cannot filter the dim.

Both: filters flow in both directions. Selecting fact rows can filter the dim. This causes ambiguity when multiple fact tables share dims, and can produce wrong DAX totals. Enable deliberately, only when needed, and document why.

3. Active vs inactive relationships

Only one relationship between two tables can be active at a time. fact_InternetSales has two date columns (OrderDateKey, ShipDateKey) — both relate to dim_Date[DateKey], but only one is active. The active one is used automatically; the other is inactive (dotted line). To use the inactive one, wrap a measure in USERELATIONSHIP inside CALCULATE.

-- Sales analysed by Ship Date instead of the default Order Date
Sales by Ship Date = CALCULATE(
    SUM(fact_InternetSales[SalesAmount]),
    USERELATIONSHIP(fact_InternetSales[ShipDateKey], dim_Date[DateKey])
)
-- The active relationship (OrderDateKey → DateKey) is temporarily replaced
-- by the inactive one (ShipDateKey → DateKey) for this measure only
09

Data types & formatting — the invisible foundation

Wrong data types are the source of a surprising number of Power BI errors. They look like data problems but are actually model configuration problems.

Critical rules

Numeric columns imported from CSV default to Text. CSV files have no type metadata — Power BI guesses, and guesses wrong for financial figures with thousand separators ("1,234.56" is text). Always check and set types explicitly in Power Query.

Date columns in financial exports often arrive as text ("20231231", "31/12/2023"). Convert to Date type using M's date parsers: Date.FromText("31/12/2023", [Format="dd/MM/yyyy"]). AdventureWorks' DateKey is a Whole Number in YYYYMMDD form — keep it as a Whole Number for joining, and use FullDate for time intelligence.

Key columns used in relationships must be the same type on both sides. A Text CustomerKey on the fact cannot join to an Integer CustomerKey on the dim. Power BI will let you create the relationship, but cross-filtering silently fails.

Formatting in the semantic model

Format strings control how measures display. Set these on every measure — not just for the display, but so Copilot and Q&A describe values correctly:

-- Common format strings (Measure tools → Format)
"$#,0"             -- $1,234 (no decimals)
"$#,0.00"          -- $1,234.56 (two decimals)
"#,0"              -- 1,234 (count, no currency)
"0.0%"             -- 12.3% (one-decimal percentage)
"#,0.0 ""K"""      -- 1,234.5 K (thousands with suffix)

Common modelling mistakes — and how to spot them

⚠ Common mistake
Flattening everything into one wide table
Importing all 9 AdventureWorks CSVs and merging them into a single 6,000-row, 60-column table because "everything is in one place."

Result: Mountain-200 Silver, 42 appears thousands of times. Product fixes have to be made on every row. You can't add a second fact (e.g. fact_ResellerSales) without flattening again.
✓ Correct approach
Load as 9 tables, connect with relationships
Keep fact_InternetSales compact (just keys + measures) and let the dims describe their entities. Snowflake chains stay normalised.

Result: tiny memory footprint, fast queries, easy to extend with new facts.
⚠ Common mistake
No date table — using OrderDateKey directly
Building time intelligence measures off fact_InternetSales[OrderDateKey] without a marked date dimension.

Result: Power BI's hidden auto date/time bloats the model; DATESYTD and similar functions behave inconsistently.
✓ Correct approach
Use dim_Date and mark it as the date table
Load dim_Date (provided in the CSV pack), Table tools → Mark as date table → FullDate. Disable auto date/time in Options.

Result: time intelligence works predictably; you control fiscal vs calendar year.
⚠ Common mistake
Bidirectional relationships everywhere
Setting filter direction to "Both" on every relationship "to make filtering work in all directions."

Result: slow performance, ambiguous filter paths in DAX, wrong totals when two facts share a dim.
✓ Correct approach
Single direction by default
Keep all relationships Single (dim → fact). Enable Both only for specific measures using CROSSFILTER() in CALCULATE, and document why.

Result: predictable filters, correct totals, better performance.
