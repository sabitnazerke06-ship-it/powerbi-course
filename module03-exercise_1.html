Exercise 03.01 — Foundation measures

The finance team needs a set of base measures before any dashboard can be built. Every financial report at Contoso starts with the same five numbers: total revenue, total cost, gross profit, gross margin percentage, and transaction count. You will create all five as explicit measures stored in the _Measures table.

Expected Results

Place each measure in a Card visual with no filters active. You should see exactly the following values:

The following columns already exist in the model and may be referenced in your solution:

ObjectTypeDescription
Sales[SalesAmount]ColumnTotal revenue value per transaction row

Tasks

1
Create a measure named Total Revenue in the _Measures table that sums all sales amounts.
HintUse the aggregation function that sums a numeric column. The measure should respond to whatever filter context is active in the report.
2
Create a measure named Total Cost using the same pattern.

Write it as a separate measure — you will reference it in the next task. A measure that references another measure updates automatically if the underlying logic changes.

3
Create a measure named Gross Profit by subtracting Total Cost from Total Revenue.
HintReference the two measures you already created — do not reference the raw columns directly. Use square brackets: [Total Revenue] - [Total Cost].
4
Create a measure named Gross Margin % expressing gross profit as a percentage of total revenue. Format it as a percentage with 1 decimal place.
HintDivision in DAX can return errors when the denominator is blank or zero. Use the DAX function designed to handle this safely — not the / operator.
5
Create a measure named Total Transactions that counts the number of rows in the Sales table.
HintUse COUNTROWS rather than COUNT on a column. COUNT skips rows where the column value is blank, which can silently understate your transaction count if any rows have data quality issues.
Solution — click to reveal ▼
Measure: Total Revenue — in _Measures table
Total Revenue = SUM(Sales[SalesAmount])
Measure: Total Cost — in _Measures table
Total Cost = SUM(Sales[TotalCost])
Measure: Gross Profit — in _Measures table
Gross Profit = [Total Revenue] - [Total Cost]
Measure: Gross Margin % — in _Measures table
Gross Margin % = DIVIDE([Gross Profit], [Total Revenue])

Format: select the measure in Fields panel → Column Tools → Format → Percentage → 1 decimal place.

Measure: Total Transactions — in _Measures table
Total Transactions = COUNTROWS(Sales)

COUNTROWS counts every row regardless of column values. COUNT(Sales[SalesAmount]) would skip rows where SalesAmount is blank — which can happen with data quality issues — silently understating the transaction count.

Exercise 03.04 — Working backwards from a requirement

The CFO asks: "I want to see how our gross margin percentage has changed month by month. I need to understand whether we are getting more or less efficient over time."

This exercise does not give you a step-by-step guide. You must decompose the requirement yourself — identify what measure, dimension, filter and visual type are needed — and then build it.

Expected Results

A line chart showing Gross Margin % on the Y-axis across all months of the year, with each year displayed as a separate coloured line. Values fall in the 56% – 59% range with visible seasonal variation.

Tasks

1
Decompose the CFO's requirement into four components before opening any Power BI panel: measure, dimension, filter, visual type.
HintWrite your answers on paper first. Measure = what number is tracked. Dimension = what drives the X-axis. Filter = is there a restriction on the data? Visual type = what chart type best shows change over time?
2
Build the line chart. Configure X-axis, Y-axis, and Legend fields. Make sure months appear in numeric order (1–12), not alphabetically.
HintIf months appear alphabetically (April, August…): in the Fields panel, find Calendar[MonthName] → Column Tools → Sort by Column → select Calendar[MonthOfYear]. This fixes the sort order without changing what is displayed.
Solution — click to reveal ▼
Step 1 — Decomposition

Measure: Gross Margin % — already exists from 03.01, no new DAX needed.

Dimension: Calendar[MonthOfYear] on X-axis, Calendar[Year] as legend so each year is a separate line.

Filter: None required — show all years and all months.

Visual type: Line chart — always the correct choice when the question is "how does X change over time?"

Step 2 — Visual configuration
  • Visual type: Line chart
  • X-axis: Calendar[MonthOfYear]
  • Y-axis: [Gross Margin %]
  • Legend: Calendar[Year]
  • Sort fix: Calendar[MonthName] → Sort by Column → Calendar[MonthOfYear]
No DAX required — key insight

Gross Margin % already responds to whatever filter context is active. When the visual puts Year in the Legend and MonthOfYear on the X-axis, each cell of the visual has a unique filter context (e.g. Year=2008, Month=3). The measure evaluates in that context automatically — no modifications needed.

★ Bonus
Spot and fix the implicit measure

A colleague built a quick visual during a stakeholder meeting by dragging Sales[SalesAmount] directly into a bar chart — no measure was created. The visual shows correct numbers, but it is wrong for a production report.

Tasks

1
Reproduce the problem: add a bar chart, drag Sales[SalesAmount] directly into Values (do not use your Total Revenue measure). Set axis to ProductCategory[ProductCategory]. Observe what appears in the Values field.

The field shows "Sum of SalesAmount" — this is Power BI auto-generating an implicit measure. It is not stored in the model, it has no name you chose, and it cannot be reused.

2
Fix it: replace the implicit measure with your Total Revenue explicit measure. Verify the numbers are identical.
HintRemove the column from the Values field, then drag in [Total Revenue] from the _Measures table. Numbers will match — but now the label reads "Total Revenue" and the logic is stored in the model permanently.
Why this matters — click to reveal ▼
The three problems with implicit measures

1. Not reusable. An implicit measure lives inside one visual only. Every new visual requires you to recreate it. An explicit measure can be dragged into any visual on any page.

2. Not auditable. If a client asks "how is Total Revenue calculated?", you cannot show them. The logic is buried in a visual, not visible in the Fields panel.

3. Name is auto-generated. "Sum of SalesAmount" is not a business name. It exposes internal column names to report users. "Total Revenue" communicates what the number means.
