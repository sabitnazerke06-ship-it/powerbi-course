01

Two contexts, one evaluation engine

Every DAX expression evaluates inside a context. Context determines which rows of data the expression "sees" when it calculates. There are exactly two types of context in DAX, and the difference between them is the key to understanding everything else.

The Excel analogy — before anything else. You already know both contexts from Excel, just by different names. Row context is what happens when you write =B2*C2 in a spreadsheet cell — Excel is "standing on row 2" and reads values from that specific row. Filter context is what a PivotTable does — it takes the full dataset and shows only the rows that match your selected Region, Year, and Category. DAX works identically, just more powerfully.
Row context vs filter context — a direct comparison

The critical insight: measures always evaluate in filter context, never in row context. When you write SUM(FactSales[SalesAmount]), DAX sums all rows that survive the current filter context — whatever filters are active from slicers, visual axes, and page filters. Calculated columns, on the other hand, always evaluate in row context, seeing one row at a time.

02

Row context in detail

Row context is the simpler of the two. When DAX iterates over a table row by row — during a calculated column evaluation or inside an iterator function — it maintains a row context. In that state, you can reference column values from the current row directly.

Contoso dataset
-- Calculated column on FactSales
-- DAX creates row context automatically: iterates every row
Gross Margin = FactSales[SalesAmount] - FactSales[TotalCost]
-- Row 1: SalesAmount=500, TotalCost=320  → Gross Margin=180
-- Row 2: SalesAmount=750, TotalCost=480  → Gross Margin=270
-- Each row is evaluated independently — this is row context
The row context trap: SUM inside a calculated column. Inside a calculated column, if you write SUM(FactSales[SalesAmount]), it does NOT sum only the current row. SUM ignores row context entirely and sums the entire unfiltered table — you get the same grand total on every single row. This is one of the most common and confusing mistakes in early DAX work. If you need the value of the current row, reference the column directly: FactSales[SalesAmount]. If you genuinely need a filtered sum, wrap it in CALCULATE (explained in Part 4).

Wrong — same value on every row

TotalQty = SUM(FactSales[Quantity])

Result: 43,517 on every row. SUM ignores row context and aggregates the entire table.

Correct — row value only

RowQty = FactSales[Quantity]

Result: the actual quantity for that specific row. No aggregation needed — row context provides the value directly.

03

Filter context in detail

Filter context is what makes Power BI interactive. Every time a user clicks a slicer, selects a bar in a chart, or applies a page-level filter, they are modifying the filter context. Measures evaluate inside whatever filter context currently exists. Filter context is not a single thing — it is built from multiple layers that stack together:

Where filter context comes from — layers stack top to bottom

All four layers are active simultaneously when a measure evaluates in a visual. Each layer narrows the set of rows. The measure sees only rows that satisfy all active conditions at once — this is filter propagation through the star schema, where dimension table filters flow through relationships into the fact table.

04

CALCULATE — the context modifier

CALCULATE does one thing: it evaluates an expression in a modified filter context. You provide an expression and one or more filter arguments. CALCULATE takes the current filter context, applies your modifications, evaluates the expression in the resulting context, and returns the result.

-- CALCULATE syntax
CALCULATE(
    <expression>,    -- what to calculate
    <filter1>,       -- modification 1
    <filter2>,       -- modification 2 (AND logic — both apply)
    ...
)
CALCULATE replaces, it does not add. This is the single most important thing to internalise. If the current context has ChannelName = "Store" from a slicer and your CALCULATE specifies ChannelName = "Online", the result is Online only — not both. The filter argument replaces the existing filter on that column. Filters on other columns are unaffected and remain unchanged.
Contoso dataset
-- Base measure
Total Revenue = SUM(FactSales[SalesAmount])

-- CALCULATE: ignores any channel slicer, always returns Online only
Online Revenue = CALCULATE(
    [Total Revenue],
    DimChannel[ChannelName] = "Online"
)

-- Multiple filters: AND logic — both conditions apply simultaneously
US Store Revenue = CALCULATE(
    [Total Revenue],
    DimChannel[ChannelName] = "Store",
    DimGeography[RegionCountryName] = "United States"
)

-- Full-price sales only (no promotions)
Full Price Revenue = CALCULATE(
    [Total Revenue],
    DimPromotion[PromotionType] = "No Discount"
)

05

Table functions — scalar vs table results

Most DAX functions return a single number — a scalar. For example, SUM(FactSales[SalesAmount]) returns one number: the total. Table functions are different — they return an entire table: a set of rows and columns.

You cannot place a table directly into a report visual. Instead, table functions are passed as arguments to other functions. The most common pattern: pass a table function into CALCULATE to modify filter context, or into an iterator (like SUMX) to define which rows to iterate over.

The table functions you will use this week
FunctionWhat it returnsPrimary use
ALL(table) All rows of the table, ignoring all filters CALCULATE argument to remove filters; or standalone to return all rows
ALL(table[col]) All unique values of a column, ignoring filters Remove filter on one column only, leaving others intact
REMOVEFILTERS(table) Same as ALL — removes all filters from a table CALCULATE argument only. More explicit than ALL when intent is to clear filters
ALLEXCEPT(table, col1, col2…) All rows with filters removed — except on specified columns Keep some filters, remove the rest. CALCULATE argument
FILTER(table, condition) Rows from table where condition is true Add conditions to context — especially for >, <, <> comparisons
SUMMARIZE(table, col1, col2…) Unique combinations of specified columns — like SQL GROUP BY Grouping and aggregation; follows relationships across tables
ALL inside vs outside CALCULATE behaves differently. Used as a CALCULATE argument, ALL removes filters — it tells the engine "ignore whatever filter is on this table." Used standalone (outside CALCULATE), ALL returns all rows of a table as a physical result. This dual behaviour is one of the most common points of confusion for new DAX writers. When you see ALL in a CALCULATE argument, always read it as "remove filter on this."
06

ALL, REMOVEFILTERS and ALLEXCEPT

ALL — seeing the full picture

The most common use of ALL is computing a percentage against a grand total. The challenge: when a report filters to one category, every measure in that visual automatically sees only that category's rows. The denominator of a percentage must ignore that filter — otherwise every row shows 100%.

Start with two measures. The first simply sums revenue — it responds to any active filter. The second uses ALL to always return the grand total regardless of what is selected:

Contoso dataset
Total Revenue = SUM(Sales[SalesAmount])

All Revenue = CALCULATE([Total Revenue], ALL(ProductCategory))

Here is what both measures show when placed in a report sliced by product category. Notice that All Revenue shows the same number — $8.34B — on every row, completely ignoring the category filter:

ALL on a column — % of parent category

When you pass a column (not a full table) to ALL, it removes the filter on that column only, leaving all other filters intact. This enables the "% of parent" pattern — each subcategory shows its share within its own category, not across the entire company.

-- ALL on the full table: removes ALL product filters → % of grand total
Revenue % of Total = DIVIDE(
    [Total Revenue],
    CALCULATE([Total Revenue], ALL(ProductCategory, ProductSubcategory))
)

-- ALL on one column: removes only subcategory filter → % of parent category
Revenue % of Category = DIVIDE(
    [Total Revenue],
    CALCULATE([Total Revenue], ALL(ProductSubcategory))
)

REMOVEFILTERS — same as ALL, clearer intent

REMOVEFILTERS is technically identical to ALL when used as a CALCULATE argument. It exists purely for readability. Compare these two lines — they produce the same result, but the second makes the intent immediately obvious to anyone reading the code:

-- Works, but the intent is ambiguous — is ALL here to return a table or remove a filter?
CALCULATE([Total Revenue], ALL(Channel))

-- Intent is unambiguous: this is explicitly removing the Channel filter
CALCULATE([Total Revenue], REMOVEFILTERS(Channel))

Both produce the same number. Use REMOVEFILTERS when writing code that others will maintain — it makes the purpose of the filter argument self-documenting.

ALLEXCEPT — remove most filters, keep a few

ALLEXCEPT is the inverse of ALL. Instead of listing the filters you want to remove, you list the ones you want to keep. Every other filter on the table is removed.

A typical use case: you want the denominator to be the total for the current channel, regardless of any product category filter. Keeping the channel filter while removing the product filter is exactly what ALLEXCEPT does:

-- % of channel total — ignores category filter, keeps channel filter
Revenue % of Channel = DIVIDE(
    [Total Revenue],
    CALCULATE(
        [Total Revenue],
        ALLEXCEPT(Channel, Channel[ChannelName])
    )
)

Choosing between ALL and ALLEXCEPT — decision guide

Use ALL(Table) when the denominator should be completely independent of a dimension. "% of grand total" always uses ALL on all product tables.

Use ALL(Table[Column]) when you want to remove the filter from one column only, while other columns in the same table remain filtered. Classic "% of parent" pattern.

Use ALLEXCEPT(Table, col1, col2…) when you have many columns and it is cleaner to declare what to keep rather than what to remove. If you need to remove 5 out of 6 columns, ALLEXCEPT(Table, kept_col) is simpler than ALL(col1, col2, col3, col4, col5).

Use REMOVEFILTERS instead of ALL when the intent of your code is to remove a filter — not to return a table. Identical result, clearer code.

07

FILTER — adding conditions to the context

Unlike ALL and REMOVEFILTERS which remove filters, FILTER adds a condition. It evaluates a table row by row, returns only the rows satisfying a condition, and that restricted table becomes the new context inside CALCULATE.

FILTER for comparisons, direct equality for simple matches. For ChannelName = "Online", use the direct syntax — it is significantly faster because Power BI can optimise it without physically scanning rows. Reserve FILTER for conditions that require comparison operators (>, <, <>, IN) or filtering on a calculated value. Using FILTER where a direct equality would work is a common performance antipattern.
Contoso dataset
-- FILTER needed: comparison operator (>)
Premium Product Revenue = CALCULATE(
    [Total Revenue],
    FILTER(DimProduct, DimProduct[UnitPrice] > 500)
)

-- FILTER needed: filtering on a calculated value
High Value Customer Revenue = CALCULATE(
    [Total Revenue],
    FILTER(DimCustomer, DimCustomer[YearlyIncome] > 80000)
)

-- No FILTER needed: simple equality — use direct syntax instead
Online Revenue = CALCULATE(
    [Total Revenue],
    DimChannel[ChannelName] = "Online"   -- faster than FILTER
)

A practical FILTER pattern from real models — the "good customers" count. This requires counting customers who made 2 or more purchases. Without FILTER + CALCULATE, the count is wrong:

Wrong — same number on every row

FILTER(DimCustomer, COUNTROWS(FactSales) >= 2)

FILTER iterates DimCustomer row by row, but COUNTROWS(FactSales) has no connection to the current customer — it returns the full FactSales row count (e.g. 10,000) for every customer, so all customers appear to qualify.

Correct — CALCULATE creates the link

FILTER(DimCustomer, CALCULATE(COUNTROWS(FactSales)) >= 2)

CALCULATE triggers context transition (see Part 10): the current DimCustomer row becomes a filter on FactSales. Now COUNTROWS counts only that customer's transactions. The condition works correctly.

08

SUMMARIZE, VALUES, DISTINCT — and the blank row

SUMMARIZE is the DAX equivalent of SQL's GROUP BY. It takes a table, returns only the unique combinations of the specified columns, and can follow relationships to group by columns from related tables.

Contoso dataset
-- Group Sales by Product Category — follows the relationship Sales → ProductCategory
SUMMARIZE(Sales, ProductCategory[ProductCategory])

-- Correct pattern: ADDCOLUMNS wraps SUMMARIZE to add measures safely
ADDCOLUMNS(
    SUMMARIZE(Sales, ProductCategory[ProductCategory]),
    "@Revenue", [Total Revenue]
)
Golden rule: never add measures inside SUMMARIZE. The safe and correct pattern is always ADDCOLUMNS wrapping SUMMARIZE. ADDCOLUMNS adds the measure evaluation outside of SUMMARIZE's internal grouping logic, giving correct results every time.

ALL, VALUES and DISTINCT — three ways to see a column

These three functions all return a table of column values, but they differ in one crucial way: how they respond to filter context and whether they include the blank row. Understanding the difference is essential for writing measures that produce correct totals.

Num All Colors    = COUNTROWS(ALL(Product[Color]))
Num Value Colors  = COUNTROWS(VALUES(Product[Color]))
Num Distinct Colors = COUNTROWS(DISTINCT(Product[Color]))

When all relationships in the model are valid, VALUES and DISTINCT return identical results. The difference only appears when the model has an invalid relationship — which triggers creation of the blank row. Here is what the report looks like when all relationships are intact:

Where the blank row comes from

DAX enforces referential integrity. When a row in a fact table references a key that does not exist in the dimension — for example, Sales rows with a ProductKey that no longer exists in Product — those orphaned rows cannot disappear (their revenue would vanish from totals). DAX's solution: it automatically adds one blank row to the dimension table and assigns all orphaned fact rows to it.

Invalid relationship — DAX creates one blank row in the dimension

Only one blank row is ever created, regardless of how many orphaned keys exist. All invalid Sales rows are assigned to it.

VALUES vs DISTINCT — when to use which

VALUES includes the blank row. Use VALUES when iterating over a dimension table with SUMX or FILTER — this ensures orphaned fact rows contribute to the result and your totals remain consistent.

DISTINCT ignores the blank row. Use DISTINCT when you explicitly want to exclude unmatched data from a count or iteration.

In a clean model (all relationships valid, no orphaned keys) VALUES and DISTINCT return identical results. The difference only surfaces with bad data.

Rule: always use VALUES when iterating over dimension tables in SUMX. If your totals are inexplicably smaller than expected, a blank row is the first thing to investigate.

The blank row in SUMX — a concrete example

Here is the most common way the blank row causes incorrect results in production models. Suppose you want to calculate revenue with tax added per product, using a Tax Rate column stored in the Product table:

Wrong — orphaned sales disappear

Sales Plus Tax = SUMX('Product', [Total Revenue] * (1 + Product[TaxRate]))

Iterates the physical table. The blank row is excluded — orphaned sales (e.g. $2.9M) do not contribute. Total can be lower than Total Revenue — mathematically impossible since tax is always positive. This is a silent error: no warning, no message.

Correct — VALUES includes orphaned sales

Sales Plus Tax = SUMX(VALUES('Product'), [Total Revenue] * (1 + Product[TaxRate]))

VALUES includes the blank row. For that row, TaxRate is blank (treated as 0), so the calculation returns Sales Amount × 1 = Sales Amount. Orphaned sales are added at their base value. Total is now larger than Total Revenue — which is correct.

The root cause is always the model, not the DAX. A blank row means your data has referential integrity issues — Sales rows with keys that don't exist in the dimension. The permanent fix is to clean the source data. VALUES is the workaround that keeps your calculations correct while the data issue exists.

These patterns cover the majority of what CALCULATE is used for in production models. Each one solves a recurring business requirement.

Context transition — the advanced concept

Read this section twice. Context transition is the most conceptually difficult topic in this lesson. If it does not fully click on the first read, that is normal. The exercise will make it concrete. What matters right now is knowing it exists and recognising it when you see it — the full implications emerge in Week 6.

Context transition is a specific behaviour that happens when CALCULATE is used inside a row context — inside a calculated column or inside an iterator function like SUMX or FILTER. When CALCULATE is called in this situation, it performs a transformation: it takes every column value of the current row and converts them into filter context.

In plain terms: CALCULATE looks at the current row, sees all its column values, and creates a filter that shows only rows matching every one of those values throughout the entire data model. The filter then propagates through relationships — so other connected tables also become filtered to match that row.

Context transition — step by step
1
Iterator is active
SUMX or FILTER is iterating DimProduct. It is currently on the row for ProductKey = 1707 (Flight Simulator, Brand: Tailspin, Color: Black). Row context is active.
2
CALCULATE is called
Inside the iterator, CALCULATE is encountered. This triggers context transition.
3
Row context → filter context
CALCULATE converts the current row into a set of filters: ProductKey = 1707 AND Brand = "Tailspin" AND Color = "Black" AND … (all columns of the current row).
4
Filter propagates through relationships
The filter flows from DimProduct to FactSales through the model relationship. FactSales now shows only sales rows for this specific product.
5
Expression evaluates on filtered rows
The expression inside CALCULATE — for example, SUM(FactSales[SalesAmount]) — now sees only this product's sales. The next iteration moves to the next DimProduct row, and the process repeats.
-- Calculated column in DimProduct
-- For each product row: how much total revenue did this product generate?

-- WRONG: SUM ignores row context — returns the same grand total on every row
Product Revenue Wrong =
    SUMX(FactSales, FactSales[SalesAmount])

-- CORRECT: CALCULATE triggers context transition
-- current DimProduct row → filter on FactSales → only this product's rows
Product Revenue =
    CALCULATE(SUM(FactSales[SalesAmount]))

-- Important: referencing a measure triggers implicit context transition
-- [Total Revenue] inside an iterator = CALCULATE([Total Revenue])
-- The CALCULATE is invisible but always present when you reference a measure
Measure reference = implicit CALCULATE. Every time you reference a measure by name — [Total Revenue] — inside a row context (iterator or calculated column), DAX automatically wraps it in CALCULATE. This means context transition happens implicitly whenever you use a measure inside SUMX or FILTER. You do not need to write CALCULATE explicitly unless you also need to pass filter arguments.
