Fact table columns (go in fact_Orders): Row ID, Order ID, Order Date, Ship Date, Customer ID, Product ID, Sales, Quantity, Discount, Profit.

Dimension table columns:
• dim_Customers: Customer ID, Customer Name, Segment
• dim_Products: Product ID, Category, Sub-Category, Product Name
• dim_Region: Country, City, State, Postal Code, Region

Tricky ones:
• Ship Mode — dimension attribute (describes the shipment method). It could be its own dim_ShipMode or added to the fact table as a category column. For this course we keep it in the fact table as a categorical column since it describes the order, not a separate entity.
• Row ID — fact table primary key (PK) — surrogate identifier for each row, not meaningful beyond that.
• Order ID — technically a degenerate dimension (it lives in the fact, not a separate dimension table). Multiple rows can share the same Order ID (one order, multiple line items).
✓
Identify the grain of the fact table
What does one row represent? Be as specific as possible in one sentence.
⌄

Write your grain statement below. This is the most important sentence you will write about this dataset.

Grain: One row in fact_Orders represents one line item within one order — meaning one product in one shipment to one customer on one date. A single order (Order ID) can have multiple rows if it contains multiple products.

Why this matters: the grain tells you what Sales means in context. Sales is the revenue for that product in that order, not the total order value. When you sum Sales across multiple rows with the same Order ID, you get the full order value. Getting the grain wrong leads to double-counting errors.
✓
Classify each numeric column: additive, semi-additive, or non-additive
Additive = safe to SUM in all directions. Semi-additive = sum across some dimensions but not all. Non-additive = never sum.
⌄

This matters for DAX. If you SUM a non-additive measure, you get a wrong number with no error message. Pick a classification and write a short reason for each.

Sales: Additive. You can sum it across products, time, regions, customers — any combination. Total sales for the East region in Q3 is just a SUM.

Profit: Additive. Same as Sales — summing profit across any dimension gives a meaningful total.

Quantity: Additive. Total units sold is a valid aggregation in any direction.

Discount: Non-additive. This is the tricky one. Discount is a rate (0 to 1). Summing discounts across rows gives a meaningless number. A 20% discount on a $100 order and a 20% discount on a $10 order are not "40% discount combined." You calculate average discount with AVERAGE() or weighted average — never SUM. This is a critical pattern you will apply in DAX from Module 3 onward.
Phase 02 Spot the modeling errors 3 tasks

Recognising a bad model before you build it — or when reviewing a colleague's work — is one of the most valuable skills you can develop. These scenarios are all based on real mistakes.

✓
Scenario A — The flat table report
Read the scenario and identify all the modeling problems.
⌄

A colleague built a Power BI report on a single flat table with all 21 columns loaded directly — no Power Query transformations, no dimension tables, no relationships. They show you the report: "It works! I can see sales by category."

List at least three problems that will appear when this report is used in production:

Problems that will appear:

1. Incorrect cross-table filters. When a user adds a slicer from what should be a dimension (e.g., Region) and a measure from what should be a fact (e.g., Sales), the filter works — but it is working on a flat table. There is no relationship model, so DAX measures cannot correctly handle complex filtering across "entity" boundaries.

2. Redundant data in memory. "Furniture" is repeated ~5,000 times in the Category column. In a proper dim_Products table, it appears once. The flat table wastes significant memory on repeated text strings. At enterprise scale this causes slow refresh and slow queries.

3. No single source of truth for product attributes. If Product Name needs to be corrected (a typo, a rebranding), you must update every row that contains it — potentially thousands. In a dimension table you update one row.

4. Cannot relate to other data sources. You cannot join this flat table to another dataset (a budget table, a returns table) without recreating the flat structure. A star schema joins via keys.
✓
Scenario B — The many-to-many trap
Spot why this model is wrong and what the fix is.
⌄

A colleague tries to create a relationship between fact_Orders[Region] (text: "East", "West", etc.) and a new dim_Region table that also has a Region column. Power BI warns about a many-to-many relationship.

Explain why this warning appears, and what the correct fix is:

Why the warning appears: "East" appears many times in fact_Orders[Region] (thousands of orders are in the East region). If dim_Region also has a Region column used as the key, "East" appears there too — potentially once, but Power BI checks the "one" side for uniqueness. If dim_Region was built by just deduplicating the Region column from the flat file, it has 4 rows (East, West, Central, South) — Region is unique there. The warning may appear because the column used to join is text, not an ID — Power BI may detect ambiguity.

The correct fix: Create a proper surrogate key (RegionID — an integer) in dim_Region using an Index column. Then add RegionID to fact_Orders via a merge. Now the relationship joins on an integer key that is unique on the dim side (each RegionID = one region) and appears many times on the fact side. Clean 1:M relationship, no warning.
✓
Scenario C — Wrong filter direction
A slicer on products is not filtering sales. Why?
⌄

An analyst creates a relationship between fact_Orders and dim_Products, but accidentally sets the cross-filter direction to Both (bidirectional). Everything appears to work — the slicer filters the fact table correctly. But they notice that slicing fact_Orders by Category also affects a completely unrelated dim_Customers table, showing only customers who bought that category.

Is this a problem? When would bidirectional filtering cause incorrect totals?

Yes, this is a problem. Bidirectional filtering allows the fact table to filter dimension tables — the opposite of the intended direction. This creates two issues:

1. Unexpected filtering of unrelated dimensions. As described — filtering by product category now shows only customers who bought that category. This is rarely the intended behaviour and can confuse report users who see customer counts change when they filter products.

2. Incorrect totals in models with multiple fact tables. If a second fact table (e.g., fact_Returns) also joins to dim_Products, bidirectional filtering can cause the dim to simultaneously filter both fact tables through each other, producing circular filter issues and wrong aggregations.

Rule: Always use single-direction filtering (from dim → fact) unless you have a specific, documented reason for bidirectional. The default should always be Single.

Write three questions this model can answer — and one it cannot

This exercise forces you to think like the report consumer, not the model builder.

This model CAN answer:
1. What are total sales and profit by product category for the West region in 2020?
2. Which customer segment generates the highest profit margin?
3. What are the top 10 products by revenue in the Technology category?

This model CANNOT answer (yet):
1. How does this year's sales compare to the same period last year? (No date dimension → no time intelligence. Module 5 fixes this.)
2. What is the return rate by product? (No returns fact table in this model.)
3. What is the budget vs actual variance? (No budget table.)

Notice: the model defines the boundary of questions you can ask. Before building any dashboard, check that the questions the stakeholder needs are answerable with the data in the model.
