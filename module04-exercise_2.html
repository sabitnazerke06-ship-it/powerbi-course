Exercise 04.01 — Channel comparison

The Commercial Director wants to compare all four sales channels side by side in one visual — Store, Online, Reseller, and Catalog — always showing each channel independently. Without CALCULATE, a channel slicer would collapse the view to a single channel. With CALCULATE, each measure is hardwired to its own channel regardless of any channel filter active on the page.

Expected Results

A Clustered bar chart showing all four channels simultaneously. The key behaviour: adding a Channel slicer and selecting "Online" should NOT collapse the chart to one bar — all four channel measures remain visible because each one overrides the channel filter independently.

Channel revenue — no filters active
ChannelRevenue% of Total

Reseller1,104,548,700.8713.2%

Note: A ProductCategory slicer WILL affect channel measures — if you select "Computers", all four channel measures show Computers-only revenue. This is expected and correct. CALCULATE fixes the channel filter only, not the product filter. Removing the product filter is covered in Exercise 04.02.

Prerequisites

ObjectTypeDescription
[Total Revenue]MeasureCreated in Module 3 — base measure to wrap in CALCULATE
Channel[ChannelName]ColumnValues: "Store", "Online", "Reseller", "Catalog"

Tasks

1
Create four measures — Store Revenue, Online Revenue, Reseller Revenue, and Catalog Revenue — each fixing the result to its specific channel using CALCULATE.
HintEach measure follows the same pattern: wrap [Total Revenue] in CALCULATE and add a filter argument that sets ChannelName to the specific value. CALCULATE will replace whatever channel filter is currently active in the report.
2
Build a Clustered bar chart with all four channel measures and ProductCategory[ProductCategory] as a slicer. Compare the behaviour of channel measures vs [Total Revenue] when you select "Computers."

Add a Card visual with [Total Revenue] (no CALCULATE) next to the bar chart. Then select "Computers" in the slicer and observe:

  • [Total Revenue] card changes — it now shows only Computers revenue (~$3.2B). This is expected — no CALCULATE override, so the slicer affects it.
  • Channel measure bars do NOT change — Store Revenue still shows $4.85B, Online $1.69B, etc. The CALCULATE filter argument fixes ChannelName regardless of what the slicer says.

This is the core difference: measures without CALCULATE respect all active filters. Measures with CALCULATE can selectively override specific filters while leaving others intact.

Solution — click to reveal ▼
Measures — all in _Measures table
Store Revenue =
CALCULATE([Total Revenue], Channel[ChannelName] = "Store")

Online Revenue =
CALCULATE([Total Revenue], Channel[ChannelName] = "Online")

Reseller Revenue =
CALCULATE([Total Revenue], Channel[ChannelName] = "Reseller")

Catalog Revenue =
CALCULATE([Total Revenue], Channel[ChannelName] = "Catalog")
Why the slicer has no effect

The CALCULATE filter argument Channel[ChannelName] = "Store" replaces any existing filter on ChannelName in the current context. A product category slicer filters ProductCategory — a different table — and does not affect the Channel filter. The channel measures are therefore immune to product filtering.

Exercise 04.02 — % of total

The Category Analyst needs a matrix showing each product subcategory's revenue alongside two percentage figures: its share of the parent category's total, and its share of company-wide revenue. The two measures require different ALL arguments — this distinction is the core concept to practise.

Expected Results

A Matrix visual with Category and Subcategory as rows. The % of Category column sums to 100% within each category. The % of Grand Total shows each subcategory's share of $8.34B.

Visual preview — Matrix with Computers expanded
Key check: Within Computers, the % of Category column (34.50% + 29.07% + 15.83% + 8.78% + 8.35% + 3.46%) should sum to 100%. The % of Grand Total column for Computers subcategories should sum to 38.48% (Computers' share of total revenue).

Prerequisites

ObjectTypeDescription
[Total Revenue]MeasureModule 3 — numerator for both % measures
ProductCategory[ProductCategory]ColumnMatrix row — parent level
ProductSubcategory[ProductSubcategory]ColumnMatrix row — child level

Tasks

1
Create a measure named % of Category Revenue that shows each subcategory's share of its parent category's total revenue.
HintThe denominator removes the subcategory filter while keeping the category filter. In this Contoso file, ProductSubcategory is a separate table — use ALL(ProductSubcategory[ProductSubcategory]) (column reference) rather than ALL(ProductSubcategory) (full table), as the full-table version may not correctly remove the filter through the relationship chain.
2
Create a measure named % of Grand Total that shows each subcategory's share of company-wide revenue, regardless of any product filter.
HintUse two separate ALL arguments — one for each product table: ALL(ProductCategory), ALL(ProductSubcategory). Note: ALL(ProductCategory, ProductSubcategory) with multiple tables in a single call throws an error in this Power BI version.
3
Build a Matrix visual. Verify: % of Category sums to 100% within each category. To expand rows: click the double-down arrow ⇓ in the visual header, or Format pane → Row headers → +/− icons → On.

If % of Category shows the same values as % of Grand Total on every row, the subcategory filter is not being removed correctly. Try switching from ALL(ProductSubcategory) to ALL(ProductSubcategory[ProductSubcategory]) in your denominator.

Solution — click to reveal ▼
Measure: % of Category Revenue
% of Category Revenue =
DIVIDE(
    [Total Revenue],
    CALCULATE(
        [Total Revenue],
        ALL(ProductSubcategory[ProductSubcategory])
    )
)

Using ALL(ProductSubcategory[ProductSubcategory]) removes the filter on the subcategory column specifically. The category filter from the matrix row remains intact — denominator is the whole category total. If this still gives identical results to % of Grand Total, try REMOVEFILTERS(ProductSubcategory) instead.

Measure: % of Grand Total
% of Grand Total =
DIVIDE(
    [Total Revenue],
    CALCULATE(
        [Total Revenue],
        ALL(ProductCategory),
        ALL(ProductSubcategory)
    )
)

Two separate ALL arguments — one per table — removes all product filters. Denominator is always $8,341,224,364.83 regardless of context.

Exercise 04.03 — Conditional aggregation

The Product Strategy team wants to split revenue into two price tiers: premium products (unit price above $500) and budget products (unit price $500 and below). This requires FILTER because the condition uses a comparison operator — direct equality syntax inside CALCULATE cannot handle greater-than or less-than conditions.

Expected Results

Three Card visuals side by side. Premium Revenue plus Budget Revenue must equal Total Revenue ($8,341,224,364.83) — use this as your verification check.

Card visuals — no filters active
MeasureExpected valueCondition
Premium Revenue$3.9bnProduct[UnitPrice] > 500
Budget Revenue$4.4bnProduct[UnitPrice] ≤ 500
Premium Revenue %47.04%Premium ÷ Total Revenue
Verification: Premium + Budget8,341,224,364.83Must equal Total Revenue
Note on UnitPrice in this dataset. Product[UnitPrice] is the unit selling price per individual product, not per transaction. In this Contoso file the majority of products are priced below $500, which is why Budget Revenue (~$4.4bn) is larger than Premium Revenue (~$3.9bn) — the opposite of what you might expect from a premium-heavy retail brand.

Prerequisites

ObjectTypeDescription
[Total Revenue]MeasureModule 3 — used in CALCULATE and as denominator
Product[UnitPrice]ColumnUnit selling price per product — threshold: 500

Tasks

1
Create a measure named Premium Revenue that calculates total revenue only for products with a unit price above $500.
HintUse FILTER inside CALCULATE. FILTER scans the Product table row by row and keeps only rows where UnitPrice > 500. The result of FILTER becomes the new product filter context inside CALCULATE.
2
Create a measure named Budget Revenue for products with unit price at or below $500. Then create Premium Revenue % as a percentage of Total Revenue.
HintThe boundary value (exactly $500) must go into exactly one bucket. Use > 500 for Premium and ≤ 500 for Budget to ensure no overlap and no gap.
3
Add a fourth Card with [Total Revenue]. Confirm that Premium Revenue + Budget Revenue = Total Revenue.

If the sum does not match, check the boundary conditions on your FILTER arguments. Also verify the table name is Product (not DimProduct) and the column is Product[UnitPrice].

Solution — click to reveal ▼
Measures — all in _Measures table
Premium Revenue =
CALCULATE(
    [Total Revenue],
    FILTER(Product, Product[UnitPrice] > 500)
)

Budget Revenue =
CALCULATE(
    [Total Revenue],
    FILTER(Product, Product[UnitPrice] <= 500)
)

Premium Revenue % =
DIVIDE([Premium Revenue], [Total Revenue])
Why FILTER is needed here

CALCULATE accepts direct equality as a shorthand: Channel[ChannelName] = "Store" works without FILTER. But comparison operators (>, <, ≤, ≥, <>) are not supported in that shorthand. FILTER must be used whenever the condition is not a simple equality test.

Exercise 04.04 — Promotion analysis

The Marketing Analyst wants to understand how much revenue is generated at full price versus during promotional periods, and whether promotions produce meaningfully more revenue than full-price selling. The key challenge: one measure requires FILTER because the condition uses a not-equal operator (<>).

Check your Promotion table first. Before writing any measures, add a table visual with Promotion[PromotionName] to the report page and note the exact text used for the no-discount value. It may be "No Discount" or a variant. Remove the table visual after confirming the value.

Expected Results

Full Price Revenue + Promo Revenue must equal Total Revenue ($8,341,224,364.83). If they do not add up, the PromotionName text in your measures does not match exactly.

Card visuals — no filters active
MeasureExpected valueDescription
Full Price Revenue$2.80bnPromotionName = "No Discount"
Promo Revenue$5.54bnPromotionName ≠ "No Discount"
Promotion Uplift %98.17%(Promo − Full Price) ÷ Full Price

Insight: Promo Revenue ($5.54bn) is nearly double Full Price Revenue ($2.80bn), giving a 98% uplift. This means most Contoso sales happen during promotions — which is a significant finding for the Marketing team. A Promotion Uplift % above 0% means promotional periods generate more revenue than full-price periods.

Prerequisites

ObjectTypeDescription
[Total Revenue]MeasureModule 3 — base measure
Promotion[PromotionName]ColumnContains "No Discount" plus named promotional campaigns

Tasks

1
Create a measure named Full Price Revenue that returns revenue only for transactions with no discount applied.
HintThis is a simple equality filter — no FILTER function needed. Use the direct CALCULATE syntax: Promotion[PromotionName] = "No Discount".
2
Create a measure named Promo Revenue that returns revenue for all promotional transactions (everything except "No Discount").
HintThe condition is PromotionName ≠ "No Discount" — a not-equal comparison. This cannot be expressed as a direct CALCULATE argument. Use FILTER(Promotion, Promotion[PromotionName] <> "No Discount").
3
Create a measure named Promotion Uplift % expressing how much more (or less) revenue promotional sales generate compared to full-price sales.
HintUplift % = (Promo Revenue − Full Price Revenue) ÷ Full Price Revenue. A negative result means more revenue comes from full-price sales — which is a healthy sign in retail.
Solution — click to reveal ▼
Measures — all in _Measures table
Full Price Revenue =
CALCULATE(
    [Total Revenue],
    Promotion[PromotionName] = "No Discount"
)

Promo Revenue =
CALCULATE(
    [Total Revenue],
    FILTER(Promotion, Promotion[PromotionName] <> "No Discount")
)

Promotion Uplift % =
DIVIDE(
    [Promo Revenue] - [Full Price Revenue],
    [Full Price Revenue]
)
Direct syntax vs FILTER — when each applies

Full Price Revenue uses direct equality (PromotionName = "No Discount") — no FILTER needed, and this is faster because Power BI can optimise it without scanning each row. Promo Revenue uses <> — a comparison operator — which requires FILTER to iterate the Promotion table row by row and apply the condition.

Exercise 04.05 — SUMMARIZE

This exercise is not about building a report visual. It focuses on understanding what SUMMARIZE returns as a table function, why measures should never be placed directly inside it, and how the correct ADDCOLUMNS pattern fixes this. You will run three queries in Query View and compare the results.

Expected Results

Query 3 (the correct pattern) should return one row per country, with revenue and transaction count columns. Revenues across all countries should sum to $8,341,224,364.83. The United States should appear as the largest market.

Query 3 output — top rows by revenue (approximate)

United Stateslargest marketlargest count

Prerequisites

ObjectTypeDescription
[Total Revenue]MeasureModule 3
[Total Transactions]MeasureModule 3
Geography[RegionCountryName]ColumnCountry dimension
Query ViewToolView → Query View in Power BI Desktop

Tasks

1
Open Query View (View → Query View). Run this query and observe the output — a table of unique country/year combinations from Sales.
EVALUATE
SUMMARIZE(
    Sales,
    Geography[RegionCountryName],
    Calendar[Year]
)
ORDER BY Geography[RegionCountryName], Calendar[Year]

This is the basic use of SUMMARIZE — returning unique combinations of dimensions that exist in Sales. No measures yet.

2
Run this second query — measures placed directly inside SUMMARIZE. Note the results, then read why this pattern is dangerous.
EVALUATE
SUMMARIZE(
    Sales,
    Geography[RegionCountryName],
    "Revenue", [Total Revenue]  -- ⚠ do not use in production
)
HintThis may produce correct-looking results in simple cases — but SUMMARIZE uses an internal "clustering" algorithm that can silently return wrong numbers in complex models. It is never the right pattern.
3
Run this third query — the correct ADDCOLUMNS pattern. Verify that revenues sum to the expected total.
EVALUATE
ADDCOLUMNS(
    SUMMARIZE(Sales, Geography[RegionCountryName]),
    "@Revenue", [Total Revenue],
    "@Transactions", [Total Transactions]
)
ORDER BY [@Revenue] DESC

The @ prefix on column names is a convention indicating these columns were created inside this query — not from the model schema.

Solution — click to reveal ▼
The golden rule

Never add measures directly as arguments inside SUMMARIZE. Always use ADDCOLUMNS to wrap SUMMARIZE and add calculated columns outside of its grouping logic. The pattern is always:

ADDCOLUMNS(
    SUMMARIZE(FactTable, Dimension[Column]),
    "@MeasureName", [Your Measure]
)
Why queries 2 and 3 may look identical but aren't

In simple models with clean data, both queries can return the same numbers. The danger is in complex models with multiple relationships, calculated tables, or security filters — where SUMMARIZE's internal clustering produces subtly wrong aggregations with no error message. ADDCOLUMNS evaluates measures in a clean filter context derived from the SUMMARIZE grouping, which is always correct.

★ Bonus
ALLEXCEPT — % of continent revenue

The stakeholder wants a matrix showing each country's revenue alongside its percentage of the continent's total. The measure must stay sensitive to continent selection but ignore any country-level filter — exactly the use case for ALLEXCEPT.

Expected Results

A Matrix with Geography[ContinentName] as rows and Geography[RegionCountryName] as sub-rows. The % of Continent column must sum to 100% within each continent independently.

Task

1
Create a measure named % of Continent Revenue that shows each country's share of its own continent's total revenue.
HintThe denominator must remove all geography filters except the continent filter. Use ALLEXCEPT(Geography, Geography[ContinentName]) — this keeps the continent filter while removing all other Geography filters including the country filter.
Note on Matrix expand iconsIn newer versions of Power BI Desktop, the Matrix visual uses drill-down arrows at the top of the visual instead of +/− icons on each row. To expand all continents at once, click the double-down arrow ⇓ ("Expand all down one level") in the visual header. To re-enable row-level +/− icons: Format pane → Row headers → +/− icons → toggle On.
Solution — click to reveal ▼
Measure: % of Continent Revenue
% of Continent Revenue =
DIVIDE(
    [Total Revenue],
    CALCULATE(
        [Total Revenue],
        ALLEXCEPT(Geography, Geography[ContinentName])
    )
)

Why ALLEXCEPT and not ALL? ALL(Geography) would remove all geography filters — the denominator becomes worldwide revenue and every row shows % of global total, not % of continent. ALLEXCEPT keeps the ContinentName filter intact while removing the country-level filter. The denominator is therefore the continent's total, and within each continent the percentages sum to 100%.
