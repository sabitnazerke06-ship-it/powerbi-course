Phase 01 Set up your file and connect to the data 3 tasks

Before writing a single transformation, get the file opened and the source connected correctly. Spending 10 minutes here saves an hour of debugging later.

✓
Download the Sample - Superstore dataset and open Power BI Desktop

Click Use First Row as Headers if the column headers are showing as "Column1", "Column2".
Check that column headers are readable: Row ID, Order ID, Order Date, Ship Date, Ship Mode, Customer ID, Customer Name, Segment, Country, City, State, Postal Code, Region, Product ID, Category, Sub-Category, Product Name, Sales, Quantity, Discount, Profit.
Do not click Close & Apply yet

Stay in Power Query. We have five queries to build before loading anything into the model.

Expected state

Power Query Editor is open. Left panel shows one query: Sample - Superstore. Column headers are visible and readable. Applied Steps panel on the right shows 2–3 steps.

✓
Fix all data types on the source query
Wrong types now = silent wrong numbers in DAX later. Do this before branching.
⌄

Select each column and set the correct type. Power Query often guesses wrong — always verify manually.

Select Order Date column → Transform → Data Type → Date
Select Ship Date column → Data Type → Date
Select Sales → Data Type → Decimal Number
Select Profit → Data Type → Decimal Number
Select Discount → Data Type → Decimal Number
Select Quantity → Data Type → Whole Number
Select Postal Code → Data Type → Text (postal codes are identifiers, not numbers)
Why Postal Code = Text?

If you leave Postal Code as a number, codes like "01234" become "1234" — the leading zero is lost. Zip codes are identifiers, not quantities. Always store them as text.

Expected state

Applied Steps now includes a "Changed Type" step. The column type icons to the left of each header should show: calendar icon for dates, "1.2" for decimals, "123" for whole numbers, "ABC" for text columns.

After Phase 1 — Power Query Editor
Checkpoint: one query named Sample - Superstore, 21 columns visible, three Applied Steps: Source → Promoted Headers → Changed Type. All numeric/date columns now have correct type icons.
Phase 02 Build the fact table 4 tasks

The fact table contains transaction rows and foreign keys. Strip out everything descriptive — those columns live in dimension tables.

✓
Reference the source query to create the fact table branch
Right-click Sample - Superstore → Reference. Do not Duplicate.
⌄

In the Queries panel, right-click on Sample - Superstore and choose Reference. A new query appears with all the same columns — it starts where the source query ended.

Rename this new query to fact_Orders.

Reference vs Duplicate

Reference = "continue from here." If you later fix a type error in the source, fact_Orders inherits the fix automatically. Duplicate = copy of the full code. Fixes do not propagate. Always Reference.

✓
Keep only the fact table columns — remove everything descriptive
Select columns to KEEP using Ctrl+click, then right-click → Remove Other Columns.
⌄

In the fact_Orders query, hold Ctrl and click to select these columns:

Row ID — unique row identifier
Order ID — order identifier (degenerate dimension)
Order Date — transaction date
Ship Date — ship date
Customer ID — FK to dim_Customers
Product ID — FK to dim_Products
Sales, Quantity, Discount, Profit — the measures
City, State, Postal Code — keep these for now, temporarily. You will use them in the next phase to merge in RegionID, then remove them once that merge is done.

After selecting these columns: right-click → Remove Other Columns. This drops Ship Mode, Customer Name, Segment, Country, Region, Category, Sub-Category, Product Name — all of which live in dimension tables and are not needed as merge keys.

Why keep City, State, and Postal Code here?

These three columns are the merge key you will need in the next phase to bring RegionID into the fact table. If you remove them now, there is nothing left to join dim_Region on. They are a temporary bridge, not permanent fact table columns — you will drop them right after the merge.

Expected state

fact_Orders has exactly 13 columns: Row ID, Order ID, Order Date, Ship Date, Customer ID, Product ID, Sales, Quantity, Discount, Profit, City, State, Postal Code. Row count should be the same as the source (9,994 rows).

✓
Rename columns to clean, consistent names
Double-click any column header to rename inline.
⌄

Consistent naming across all tables makes DAX much cleaner. Rename columns by double-clicking the header:

Row ID → RowID
Order ID → OrderID
Order Date → OrderDate
Ship Date → ShipDate
Customer ID → CustomerID
Product ID → ProductID

Sales, Quantity, Discount, and Profit are already clean — leave them as is.

✓
Disable load on the source query
Right-click Sample - Superstore → uncheck "Enable Load".
⌄

The source query is a staging step — it should not create a visible table in the model. Right-click Sample - Superstore in the Queries panel → uncheck Enable Load.

The query name will appear in italics, indicating it is a staging query that is not loaded. Only your fact and dimension queries should load into the model.

Expected state

Sample - Superstore appears in italics in the Queries panel. fact_Orders is still bold (will load). Row count on fact_Orders: 9,994 rows.

After Phase 2 — Power Query Editor
Checkpoint: two queries in the panel — Sample - Superstore in italic (load disabled), fact_Orders selected. fact_Orders has 13 columns (including the temporary City, State, Postal Code merge keys) and ends with the Renamed Columns step.
Phase 03 Build the three dimension tables 6 tasks

Each dimension table follows the same pattern: Reference → keep relevant columns → Remove Duplicates → rename. You are extracting unique entity lists from the flat file.

✓
Build dim_Products — one row per unique product
Reference source → keep Product ID, Product Name, Category, Sub-Category → Remove Duplicates.
⌄
Right-click Sample - Superstore → Reference. Rename the new query to dim_Products.
Ctrl+click to select: Product ID, Product Name, Category, Sub-Category. Right-click → Remove Other Columns.
Rename: Product ID → ProductID, Product Name → ProductName, Sub-Category → SubCategory.
Select all columns (Ctrl+A or click first column, shift-click last) → Home → Remove Rows → Remove Duplicates. This reduces the table to one row per unique product.
Sort by ProductID ascending: click the ProductID column header → Sort Ascending.
Expected state

dim_Products has 4 columns and ~1,862 rows — one per unique product. Check: click ProductID header → no duplicate values should exist.

✓
Build dim_Customers — one row per unique customer
Reference source → keep Customer ID, Customer Name, Segment → Remove Duplicates.
⌄
Reference Sample - Superstore → rename to dim_Customers.
Keep only: Customer ID, Customer Name, Segment. Remove other columns.
Rename: Customer ID → CustomerID, Customer Name → CustomerName.
Select all columns → Remove Duplicates.
Expected state

dim_Customers has 3 columns and 793 rows. Each CustomerID appears exactly once. Segment values should be Consumer, Corporate, or Home Office.

✓
Build dim_Region — one row per unique location
This one is trickier — you need to add a surrogate key after deduplication.
⌄

The source file has no RegionID column — location is stored as Region, Country, City, State, Postal Code. You need to create a proper dimension with a generated key.

Reference Sample - Superstore → rename to dim_Region.
Keep only: Region, Country, City, State, Postal Code.
Rename Postal Code → PostalCode.
Select all columns → Remove Duplicates. Note this dedupes on the combination of all five columns together, not on City alone — the same city name can appear under different states or postal codes, and each of those combinations becomes its own row.
Add a surrogate key: Add Column → Index Column → From 1. Rename the Index column to RegionID. Move it to the first column position (drag the header).
Now you need to add RegionID back to the fact table

The fact table does not yet have a RegionID. See the next task.

Expected state

dim_Region has 6 columns (RegionID, Region, Country, City, State, PostalCode) and ~632 rows. RegionID starts at 1 and increments.

✓
Add RegionID to fact_Orders using a Merge
Merge fact_Orders with dim_Region on City + State + Postal Code to bring in the RegionID.
⌄

The fact table needs a RegionID column to join to dim_Region. We get it by merging.

Why not just City?

City names are not unique. "Springfield" or "Arlington" each show up in several different states, so a merge on City alone would match one fact row against multiple dim_Region rows, causing duplicated fact rows and a wrong RegionID. You need a composite key, the same combination of columns dim_Region was deduplicated on, to get one clean match per row.

Click on fact_Orders in the Queries panel.
Go to Home → Merge Queries (not "Merge Queries as New").
In the Merge dialog: top table = fact_Orders, bottom table = dim_Region. Hold Ctrl and click, in order, City, then State, then Postal Code in fact_Orders. Do the same, in the same order, in dim_Region. Join Kind = Left Outer.
A new column called "dim_Region" appears. Click the expand icon (↕) next to the column header. Uncheck all columns except RegionID. Uncheck "Use original column name as prefix."
Move the RegionID column to sit after ProductID (drag the header).
Now that RegionID is in place, the City, State, and Postal Code columns have done their job. Ctrl+click all three → right-click → Remove Columns.
Expected state

fact_Orders is back down to 11 columns (the original 10 plus RegionID). City, State, and Postal Code are gone. RegionID is populated for every row — no blanks. Row count is still 9,994.

✓
Check for nulls — validate all foreign keys are populated
Click the filter dropdown on each ID column and check if "(null)" appears.
⌄

Before loading, verify your foreign keys are complete. A null FK breaks a relationship silently.

In fact_Orders, click the filter ▾ on CustomerID. Is "(null)" in the list? If yes, investigate — a merge went wrong.
Repeat for ProductID and RegionID.
Also check: click each column header and look at the bottom bar — it shows distinct count and null count. RegionID should show 0 nulls.
Expected state

All three FK columns show 0 null values. If RegionID has nulls, the merge join key is wrong — check that you joined on all three of City, State, and Postal Code (not City alone).

✓
Close & Apply — load all four tables into the model
Home → Close & Apply. Watch the loading indicator for each table.
⌄

You are ready to load. Click Home → Close & Apply. Power Query runs all four queries (fact_Orders, dim_Products, dim_Customers, dim_Region) and loads them into the Power BI data model. Sample - Superstore is skipped because you disabled its load.

The loading indicator shows progress per table. Depending on your machine, this takes 15–60 seconds.

Expected state

You are back in Power BI Desktop. The Fields panel on the right shows four tables: dim_Customers, dim_Products, dim_Region, fact_Orders. Expand fact_Orders — you should see 11 fields with the correct icons (calendar for dates, sigma for numbers).

After Phase 3 — Power Query Editor (just before Close & Apply)
Checkpoint: five queries — Sample - Superstore (italic, disabled load), and the four that will load: fact_Orders, dim_Products, dim_Customers, dim_Region. fact_Orders has 11 cols now with RegionID merged in. Click Close & Apply when this matches.
Phase 04 Build the relationships in the model view 3 tasks

Tables loaded ≠ model built. You still need to define the relationships — the lines that connect your dimensions to the fact table. This is where the star schema comes alive.

✓
Open Model view and arrange your tables into a star
Click the Model view icon (diagram) in the left rail. Drag tables into star position.
⌄

Click the Model view icon (three rectangles connected by lines) in the left navigation rail. You should see your four tables laid out — probably in a random arrangement.

Drag fact_Orders to the centre of the canvas.
Drag dim_Products to the top.
Drag dim_Customers to the left.
Drag dim_Region to the bottom or right.

You should now see a star shape with the fact table in the centre. Power BI may have auto-detected some relationships — check what it created.

✓
Create or verify all three relationships
Drag the FK column from fact_Orders onto the PK column in each dimension. Or use Manage Relationships to verify auto-detected ones.
⌄

You need three one-to-many relationships — all pointing from dimension (one) to fact (many).

Products: drag fact_Orders[ProductID] onto dim_Products[ProductID]. Or check in Manage Relationships that this exists as 1:* from dim_Products to fact_Orders.
Customers: drag fact_Orders[CustomerID] onto dim_Customers[CustomerID].
Region: drag fact_Orders[RegionID] onto dim_Region[RegionID].

After each relationship is created, a line appears between the tables. The "1" appears on the dimension side, the "*" (many) on the fact side.

If Power BI auto-created wrong relationships

Go to Home → Manage Relationships. Delete any incorrect ones and create the correct ones manually. Wrong auto-detected relationships are a common source of incorrect totals.

Expected state

Three lines connect the fact table to its dimensions. Each line shows 1 on the dimension side and * on the fact side. Cross-filter direction should be Single (one arrow pointing from dim → fact).

✓
Hide FK columns from report view
Right-click CustomerID, ProductID, RegionID in fact_Orders → Hide in Report View.
⌄

Foreign key columns in the fact table are plumbing — they exist to join tables, not to appear in visuals. Hiding them from report view prevents users from accidentally dragging them into charts and getting meaningless results.

In Model view, right-click fact_Orders[CustomerID] → Hide in Report View. The column becomes grey.
Repeat for fact_Orders[ProductID] and fact_Orders[RegionID].
Also hide fact_Orders[RowID] — it is a row identifier, not a measure.
Expected state

In Report view, fact_Orders shows: OrderID, OrderDate, ShipDate, Sales, Quantity, Discount, Profit. The FK columns and RowID are hidden (still exist, just not visible to report builders).

After Phase 4 — Power BI Desktop · Model view
Checkpoint: Star schema with fact_Orders in centre, dims around it. Three relationships with 1 on the dim side and * on the fact side. The arrow on each line points from dim → fact (single-direction filter). FK columns (CustomerID, ProductID, RegionID) are greyed out — they're hidden in Report view.
Phase 05 Smoke test — verify the model works 2 tasks

Do not start building visuals until you have confirmed the model is correct. A fast smoke test now saves hours of debugging wrong numbers later.

✓
Build a quick sanity-check table visual
Go to Report view. Drag Category from dim_Products and Sales from fact_Orders into a table.
⌄

Click the Report view icon (bar chart) in the left rail. In the Visualisations panel, click the Table visual icon.

Drag dim_Products[Category] into the table's Columns field.
Drag fact_Orders[Sales] into the table's Columns field.
The table should show 3 rows: Furniture, Office Supplies, Technology — each with a sales total.
Add a Slicer visual. Drag dim_Customers[Segment] into it. Click "Corporate" — the table totals should change. This confirms the relationship between dim_Customers and fact_Orders is working.
Expected total Sales

All categories combined: $2,297,200.86. Technology: $836,154.03, Furniture: $741,999.80, Office Supplies: $719,047.03.

If the table shows the same number for every Category

The relationship between dim_Products and fact_Orders is not working. Go back to Model view and check the relationship direction — the single arrow should point from dim_Products to fact_Orders.
